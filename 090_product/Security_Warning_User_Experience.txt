Security Warning User Experience
Generated: 2026-03-08
Updated: 2026-03-14

Purpose
Define how LinkLook should communicate SAFE, SUSPICIOUS, and BLOCK decisions without confusing or overwhelming the user.

Principles
- plain language first
- explain the reason briefly
- avoid panic wording
- let the user understand what happened
- reserve hard blocking for high-confidence risk

Example states
SAFE (OK)
- Message: This link looks safe.
- Action: One large "Open" button. Opens page in LinkLook's built-in browser.
- Display mode (MDL-109): "always show result" (default) shows the safe screen
  briefly for reassurance; "auto-open safe links" skips the screen and loads
  directly in LinkLook. PROTECTED profile enforces always show result.
- There is no "Open in other browser" button. LinkLook IS the browser.

SUSPICIOUS (INFORM / WARN)
- Message: This link has warning signs.
- Explain one or two reasons such as new domain, hidden destination, or brand imitation.
- Action: Large "Go Back" button (dominant). Small "Open" button (opens in
  LinkLook). On WARN, "Open" requires an extra confirmation tap.
- "Check Message Context" button available (Pro feature).

BLOCK
- Message: This link appears dangerous and was blocked.
- Explain the strongest reason available.
- Action: Large "Go Back" button only. No forward path.

Browser model (updated 2026-03-14)
LinkLook is the user's primary and only browser. There is no browser-picker
setting and no redirect to an external browser. All pages open in LinkLook.
Edge cases (DigiD, iDEAL, OAuth) are handled via automatic silent system
handoff. See CLAUDE.md rules 10-11.

Special cases
- QR-origin links
- shortener links
- Universal Link / app-opening behavior
- repeat attempts on the same high-risk destination

Measurement
Track whether warnings reduce risky opens without creating excessive false alarms.
