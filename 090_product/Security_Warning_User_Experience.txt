Security Warning User Experience
Generated: 2026-03-08
Updated: 2026-03-16

Purpose
Define how LinkLook should communicate OK, INFORM, WARN, and BLOCK decisions
without confusing or overwhelming the user.

Principles
- plain language first
- explain the reason briefly
- avoid panic wording
- let the user understand what happened
- reserve hard blocking for high-confidence risk
- keep friction proportional to risk

Example states
OK
- Message: brief non-blocking toast: "Looks OK — opening"
- Action: no blocking screen; the page opens immediately in LinkLook's built-in browser
- There is no "Open in other browser" button. LinkLook IS the browser.

INFORM
- Message: This link may be unfamiliar.
- Explain one or two reasons such as new domain, hidden destination, or brand imitation.
- Action: large "Go Back" button (dominant). Small "Continue →" button (opens in LinkLook).
- "Check Message Context" button available (Pro feature).

WARN
- Message: This page shows warning signs.
- Explain the strongest reason available without overclaiming certainty.
- Action: large "Go Back" button (dominant). Very small "Continue Anyway" button.
- "Continue Anyway" requires an extra confirmation tap.
- "Check Message Context" button available (Pro feature).

BLOCK
- Message: This link appears dangerous and was blocked.
- Explain the strongest reason available.
- Action: large "Go Back" button only. No forward path.

Browser model
LinkLook is the user's primary browser. There is no browser-picker setting and
no redirect to an external browser for normal browsing. All pages open in
LinkLook. Edge cases (DigiD, iDEAL, OAuth) are handled via automatic silent
system handoff. See CLAUDE.md rules 10-11.

Special cases
- QR-origin links
- shortener links
- Universal Link / app-opening behavior
- repeat attempts on the same high-risk destination

Measurement
Track whether warnings reduce risky opens without creating excessive false alarms.
