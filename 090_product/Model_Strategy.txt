================================================================================
MODEL ARCHITECTURE
================================================================================

THREE-TIER DECISION STACK:

Tier 1 — Deterministic rules
Fast, zero-latency, on-device decisions.
Blocklist cache hits, top-500 pre-filter, structural heuristics.
No model involved. Binary outcome: BLOCK or PASS.

Tier 2 — SLM (Small Language Model) — v2+
Handles the majority of ambiguous traffic.
Target: ≥80% of non-deterministic cases resolved by SLM without a Claude call.
v1 status: slot reserved in pipeline between GSB and Claude. Empty in v1.
v2 status: trained on v1 classification records; introduced once sufficient
           training volume exists.

Tier 3 — Claude API (LLM)
Handles edge cases, uncertain cases, and cases the SLM cannot resolve.
Reserved for genuine uncertainty. Not the primary analysis engine.
Provides plain-English explanation with every verdict.
Remains the fallback for UNCERTAIN cases at v2 and beyond.

GOVERNING PRINCIPLE
No live self-modifying production models. All model updates go through the
cycle: sample → audit → validate → retrain. Version controlled. Staged
deployment.

================================================================================
V1 MODEL BEHAVIOUR
================================================================================

In v1, the SLM slot is empty. All URLs that pass the deterministic tier
(GSB + local caches) go directly to Claude.

Claude receives exactly five data points per request:
1. Full original URL including query parameters.
2. Resolved destination URL after redirect chain.
3. Domain registration date.
4. User profile: PROTECTED or STANDARD.
5. User mode: BLOCK or WARN.

Claude returns structured JSON:
  { "verdict": "SAFE" | "BLOCK" | "UNCERTAIN",
    "explanation": "one sentence in plain English" }

Explanation tone is calibrated per the four valid profile+mode combinations:
  PROTECTED + BLOCK: calm, reassuring, no jargon.
  STANDARD + BLOCK:  calm, slightly more technical detail allowed.
  STANDARD + WARN:   informative, direct, more detail allowed.
  PROTECTED + WARN:  impossible — PROTECTED profile is locked to BLOCK mode.

Claude error handling: if Claude returns an error, the verdict defaults to
UNCERTAIN with a template explanation. UNCERTAIN is then treated per user
mode (block in BLOCK mode, surface for user decision in WARN mode).

================================================================================
V2 SLM — TWO OPTIONS UNDER EVALUATION (D11)
================================================================================

DECISION DEFERRED TO V2. To be resolved once v1 training data is accumulated.

OPTION A — BACKEND SLM
Deployment: backend server.
Training:   Standard supervised learning on v1 classification records.
Advantages: Easier to train and update. No device storage impact. Can pass
            through plain-English explanations (model output → Claude-quality
            text).
Requires:   Backend call for every SLM analysis — not fully offline-capable.
Latency:    Additional network round trip for SLM cases.

OPTION B — APPLE CORE ML ON-DEVICE CLASSIFIER
Deployment: On-device, runs on Apple Neural Engine.
Training:   Create ML on v1 classification records.
Model size: 5–20MB on-device storage.
Advantages: No backend call for confident verdicts. Very low latency.
            Runs on Apple Neural Engine — battery-efficient.
Limitations: Cannot generate plain-English explanations. Confident verdicts
             use template explanations only. Claude reserved for uncertain
             cases. Model updates require app update or OTA model download.
Requires:   App update or OTA mechanism for model updates.

V1 TRAINING LOG DESIGN:
The v1 SLM training log schema has been designed to be compatible with both
options. Features captured: URL hash, domain registration date, GSB verdict,
Claude verdict, Claude explanation, hour-level timestamp. No additional
features need to be added to the v1 log to support either option.

DECISION CRITERIA FOR V2:
- Volume of accumulated training data.
- Measured quality of v1 classification records.
- Battery impact considerations for the primary audience (older devices).
- Desire for offline-capable confident verdicts (favours Option B).
- Desire for high-quality natural language explanations (favours Option A).

================================================================================
SLM TRAINING GOAL
================================================================================

Target: SLM handles ≥80% of non-deterministic URL cases without a Claude call.
The remaining ≤20% escalate to Claude for genuine uncertainty.

This target:
- Reduces Claude API cost at scale.
- Reduces latency for the majority of analysed URLs.
- Keeps Claude reserved for the cases where its reasoning adds most value.

================================================================================
LEARNING LOOP
================================================================================

v1 COLLECTION PHASE:
Every URL processed at Layer 3 emits a classification record (ADR-003).
These records form the training corpus. Must be implemented from day one.

v1.x / v2 TRAINING PHASE:
Once sufficient volume exists (~180,000 records at 12 months at target scale),
the first SLM iteration is trained on accumulated v1 records.

ONGOING CYCLE (v2+):
1. Sample a subset of live production analyses.
2. Audit: compare SLM verdict to Claude verdict; identify disagreements.
3. Validate: human review of high-disagreement cases.
4. Retrain: incorporate validated labels; version and deploy new model.

QUALITY CONTROLS:
- SLM vs Claude disagreement rate tracked as a continuous metric.
- Minimum confidence threshold required for SLM to return a verdict without
  escalating to Claude.
- Escalation rate (% of URLs reaching Claude) monitored weekly.
- False positive rate monitored weekly.

================================================================================
AI-SPECIFIC THREATS (from Security_Threat_Deep_Dive_Appendix_v1)
================================================================================

PROMPT INJECTION (LLM)
Threat:    Malicious page content attempts to manipulate the classification
           prompt, causing Claude to return incorrect verdicts.
Mitigation:
- Strict prompt isolation — page content is data input, never instruction.
  The classification prompt must be hardcoded; no part of the analysed URL
  or page content is allowed to modify the prompt structure.
- Never allow page instructions to modify classification policy.
- Deterministic validation layer before LLM output is trusted: Claude's
  JSON response is validated against expected schema before any verdict is
  acted on. Unexpected fields or formats trigger UNCERTAIN.

MODEL EVASION
Threat:    Obfuscated phishing content designed to bypass AI classification.
           Visual-only credential harvesting (e.g. login form rendered as image).
Mitigation:
- DOM + text fingerprinting alongside visual analysis at Layer 2/3.
- Heuristic pattern detection at Layer 2 before LLM involvement — catches
  structural phishing signals independent of content understanding.
- Escalation sampling for audit: a random sample of SAFE verdicts is
  flagged for human review to catch evasion patterns over time.

ZERO-DAY AND FIRST-TIME DOMAIN HANDLING
Threat:    No reputation signal exists for a first-observed domain. SLM and
           blocklist layers provide no protection.
Mitigation:
- Full sandbox render in Stage 2 for unknown domains.
- Heuristic anomaly detection (structural signals independent of reputation).
- Fingerprint storage for later correlation across similar attacks.
- Optional evidence capture under high-risk trigger conditions.
These cases are the primary reason Claude remains the fallback tier — novel
domains with no training data are exactly where LLM reasoning adds most value.

================================================================================
VENDOR STRATEGY
================================================================================

VENDOR ABSTRACTION
The model interface is abstracted behind a defined API contract. No vendor
lock-in. The SLM and LLM are replaceable components.

CLAUDE (ANTHROPIC) — CURRENT LLM
Used as the Layer 3 fallback in v1. Performance and cost to be measured
under realistic load before v2 planning.

FUTURE FLEXIBILITY
The abstraction allows switching to alternative LLM providers or running
a self-hosted model at the LLM tier without architectural changes.
