================================================================================
GOVERNING PRINCIPLE
================================================================================

LinkLook is a deterministic safety product with optional bounded on-device AI
signals. The base product does not depend on any external LLM. AI is additive
and bounded: the core safety flow must work correctly with AI disabled.

No live self-modifying production models. All model updates go through the
cycle: sample → audit → validate → retrain. Version controlled. Staged
deployment. Model updates ship with app updates (no over-the-air model
replacement in v1.x).

================================================================================
THREE-TIER DECISION STACK (current)
================================================================================

Tier 1 — Deterministic rules and policy
Fast, zero-latency, on-device decisions.
Structural analysis, Google Safe Browsing, signal extraction, policy engine.
Binary or graduated outcome: allow / inform / warn / block.
No model involved. This is the complete v1 base product.

Tier 2 — Optional on-device bounded AI signals (Pro, planned)
Two small, specialized models running on-device via Core ML:

  a) URL signal model — tiny classifier trained on URL features.
     Input: fixed-length numeric/categorical/boolean feature vector
            derived from the NormalizedURL (23 engineered features).
     Output: 3-class softmax (benign / suspicious / malicious) + confidence.
     Role: produces a signal (aiUrlRiskHigh or aiUrlClean) that enters the
           deterministic combiner. Can escalate (OK→INFORM, INFORM→WARN).
           Cannot override BLOCK. Cannot downgrade. Cannot produce verdicts.
     Runs: on every URL check (Pro users only), in parallel with Tier 1.
     Privacy: fully on-device. No data leaves the device.
     Contract: AI_Model_Contracts.md

  b) Message signal model — multilingual text classifier (Dutch + English).
     Input: raw message text + optional channel hint.
     Output: scam likelihood + scored intent labels + explanation tags.
     Role: analyzes the surrounding message text when a user taps
           "Check Message Context" on INFORM/WARN screens.
           Produces a traffic-light verdict displayed alongside (not replacing)
           the URL verdict. Never modifies the URL verdict.
     Runs: only when explicitly triggered by user action.
     Privacy: fully on-device. Text processed and discarded, never stored.
     Contract: AI_Model_Contracts.md

Tier 3 — Optional future explanation layer (deferred)
  Foundation Models (Apple) may be used later for natural-language explanation
  on supported devices only. This is an enhancement, not a requirement.
  Backend LLM remains optional and deferred — it is not architecturally
  required for any current or planned feature.

================================================================================
WHAT CHANGED FROM v1.1
================================================================================

The v1.1 document (2026-03-06) described:
  - Tier 2 as a "reserved SLM slot, empty in v1"
  - Tier 3 as "Claude API fallback — all unknown URLs go directly to Claude"
  - Claude receiving five data points and returning structured JSON
  - Claude as the active v1 runtime path for non-deterministic cases

Per the Master Decision Log 2026-03-10:
  - Core ML is the active-planned AI path (on-device, not server-side)
  - Foundation Models are optional (not required)
  - Claude backend is not architecturally required
  - The older Claude-required wording is superseded

The current architecture:
  - v1 base product runs entirely on deterministic rules + GSB
  - AI models are additive signal sources, not fallback tiers
  - No URL ever "goes directly to Claude"
  - The base product must pass all safety tests with AI disabled

================================================================================
TRAINING DATA STRATEGY
================================================================================

URL signal model:
  - Public phishing datasets: PhishTank, OpenPhish
  - Legitimate URL corpus: Tranco top-10k
  - Must validate against LinkLook's existing golden test fixtures before release

Message signal model:
  - Scam message corpora (Dutch + English)
  - Legitimate message samples
  - Must validate against AI test fixtures (see 100_testing/AI_Test_Catalog.txt)

Training log design (from v1):
  Structured evaluation records emitted by the detection engine can support
  future model training. Minimum fields: canonical URL hash, domain, timestamp,
  feature vector, triggered signals, final verdict, confidence, verdict source.
  No user identity required. Privacy by design applies to training data.

================================================================================
VENDOR STRATEGY
================================================================================

The model interface is abstracted behind defined Swift protocols
(URLSignalModelProtocol, MessageSignalModelProtocol). No vendor lock-in.
Models are replaceable components.

Core ML is the current deployment target. The protocol abstraction allows
switching to alternative runtimes without architectural changes.

Backend LLM integration (any vendor) remains possible through the same signal
interface if introduced later. It must enter as a non-authoritative signal,
not as a verdict source.

================================================================================
AI-SPECIFIC THREAT MITIGATIONS
================================================================================

Model evasion:
  Obfuscated phishing content designed to bypass AI classification is mitigated
  by the deterministic pipeline running in parallel. AI is one signal among
  many. The rule engine decides the verdict, not the AI score alone.

Zero-day and first-time domains:
  No reputation signal exists for first-observed domains. The deterministic
  engine handles this through structural heuristics, unusual TLD detection,
  and the fail-safe principle (warn on uncertainty, never silent allow).
  AI signals add coverage for subtle patterns that rules may miss.

Adversarial URLs:
  URLs crafted to fool the AI model cannot bypass the deterministic checks
  that run in parallel. The combiner's escalate-only rule means an AI miss
  does not weaken protection — it only means the AI signal is absent.
