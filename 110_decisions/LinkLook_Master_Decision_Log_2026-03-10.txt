MDL-027
Subject: Policy / Operations
Status: Decided
Title: Exact backend-client split for policy fetch and cached-policy behavior
Decision:
The backend publishes a versioned, data-only JSON policy document. The app
downloads and caches it locally. The policy contains five sections: metadata
(version, validity, signature, min app version), defaults (defaultAction for
unmatched URLs), signal weights and thresholds (numeric scoring), explicit
rules (match condition → action + reasonCode), and overrides (force-applied
decisions that always win). Evaluation order: cache lookup → overrides →
rules → score-based evaluation → defaultAction. Policy is fetched on app
launch, every foreground transition (304 or new version), and via background
refresh. Grace period: 7 days past validUntil before falling back to a
hardcoded baseline policy bundled with the app binary. Policy must sync to
the shared App Group container (MDL-105) for future Network Extension access.
The core principle: code = engine, policy = behavior. The policy is a decision
table, not executable logic. The app maps reasonCode to UX — the policy never
contains UI text or platform-specific behavior.
Full specification: 050_architecture/Backend_Policy_Specification.txt
Main source chats:
- Code Review Feedback
- LinkLook Browser Feasibility
- ChatGPT policy design conversation (2026-03-11)

MDL-100

Status: Decided

MDL-107
Subject: Architecture / Data Model
Status: Decided
Title: Three-tier URL reuse model — exact URL, pattern, domain
Target version: V1 (schema and lookup order), V2 (pattern table, smart whitelist)
Decision:
URL reuse for verdict caching and the whitelist must use a three-tier lookup
model. Each tier has a different reuse strength:

Tier 1 — Exact URL (strongest reuse):
Store the full normalized URL. Normalization includes: lowercase scheme and host,
remove default port, strip tracking parameters, strip fragments (#), normalize
query parameter order. Equivalent URLs must resolve to the same record. An exact
URL match means "this specific link was already evaluated."

Tier 2 — URL pattern (medium reuse):
Store rules per domain with path patterns and query rules. Example: domain
amazon.com + path /gp/help/* → likely safe. domain example.com + path /redirect
+ query contains target= → inspect or block. Avoids storing millions of
near-identical URLs individually. V2 feature — schema designed in V1.

Tier 3 — Domain reputation (weakest reuse):
Store the registered domain with a general reputation status (unknown / trusted /
suspicious / blocked). A trusted domain does NOT automatically approve all URLs
on that domain. This is critical — trusted.com/home being safe does not mean
trusted.com/redirect?target=evil.com is safe.

Redirect target table:
Redirect chains must be tracked separately. A safe-looking first URL that
redirects to an unsafe final destination must be caught. Store: source URL hash,
final URL hash, redirect chain, and verdict status.

Decision log (traceability):
Every decision must be reproducible. The decision record includes: input URL,
normalized URL, matched domain, matched pattern (if any), matched exact URL
(if any), final verdict, and why.

Lookup order:
1. Normalize URL
2. Check exact URL match (tier 1)
3. Check known patterns (tier 2)
4. Check domain reputation (tier 3)
5. Check redirect behaviour
6. Return final verdict

V1 KISS scope: domains, urls, and decisions tables only. This keeps the first
release simple while establishing the schema and lookup order. V2 adds
url_patterns and redirects tables.

Full schema with field definitions and examples is in
050_architecture/URL_Reuse_Data_Model.txt.

This model aligns with the existing Data Architecture tables (domain_verdict_cache,
url_verdict_cache, inspection_job) and extends them with the pattern tier and the
explicit lookup priority order. The key safety principle: domain trust alone never
approves a URL.

MDL-110
Subject: Security / Entry Path Visibility
Status: Decided
Title: Entry path visibility analysis — which URL paths reach LinkLook and
which do not
Target version: V1 (documentation and honesty), V2 (Action Extension mitigates
in-app browser blind spot), V3 (Network Extension closes most remaining gaps)
Decision:
LinkLook receives a URL only when iOS delivers it as a standard web-link
handoff to the system default browser. The following paths bypass LinkLook:

Not a browser case:
- Universal links (domain has AASA, matching app installed → app opens directly)
- Custom URL schemes (bankapp://..., maps://... → target app opens directly)
- App Clips (App Clip URL → iOS launches App Clip experience)
- Non-web QR content (Wi-Fi config, vCard, plain text → no browser involved)

Still a web URL, but another app keeps control:
- In-app browser (WKWebView / SFSafariViewController → scanning or messaging
  app shows the page internally, URL never reaches the default browser)
- Notification hardcodes a specific browser (opens via browser-specific scheme)

The most common and highest-risk case for the primary audience — a plain
https:// URL tapped in Messages or scanned with the native Camera — reaches
LinkLook. The blind spots are named limitations, documented honestly in
onboarding, the App Store description, and the founding story.

The product promise is: "LinkLook checks every web link that reaches it as
your default browser." Not: "LinkLook checks every link on your phone."

MDL-111
Subject: Architecture / Policy Schema
Status: Decided
Title: Enriched policy rule schema — per-rule validity, source, override
controls, and deterministic matching order
Target version: V2 (implemented when backend policy fetch is built)
Decision:
The policy rule schema (Backend_Policy_Specification.txt sections 2.4–2.6) is
enriched with per-rule fields that make the policy auditable, time-bounded, and
operationally controllable:

Per-rule additions:
1. validity (not_before / expires_at) — rules have a time window. Expired
   approvals fall back to the next matching rule or defaultAction, never to
   silent allow.
2. source (type + ref) — provenance tracking for audit. Values: manual_review,
   backend_dataset, partner_feed, temporary_override, crowdsourced.
3. Per-override allow_user_bypass — controls whether the UI may let the user
   continue despite the override decision.
4. Per-override sticky — prevents displacement by a later lower-confidence rule
   at the same priority.
5. notes — operator note, not used by the client engine.

Matching order (section 2.6):
Disabled and out-of-window rules are filtered first. Remaining matches are
sorted by priority descending, then by specificity (exact URL > domain + path
prefix > domain > signal), then by severity. The order is deterministic.

Priority numbering guidance: 1000 (emergency) → 500 (exact URL) → 300 (path
prefix) → 100 (domain) → 50 (signal) → 10 (informational).

Scope principle: an approved exact URL does NOT automatically approve the
entire domain. Domain-level approval requires an explicit domain-level rule.

V1 constraints: no regex, no user-specific rules, no A/B testing.
decision.message is excluded — user-facing text comes from reason code mapping,
keeping the policy language-neutral.

MDL-113
Subject: Security / Assessment
Status: Decided
Title: Security assessment — current-state evaluation with hardening
requirements
Target version: V1 (documentation, blind spot honesty), V2 (hardening
controls HR-01 through HR-09)
Decision:
LinkLook is rated "security-promising, not yet security-proven."

Strengths: minimal scope, deterministic decision flow, policy-driven
behavior, exact URL vs domain separation, fail-safe defaults, no web
content before verdict, privacy-first data separation.

Five trust boundaries identified: iOS URL handoff, backend policy

Ten threats cataloged, highest impact first: policy backend compromise,

certificate pinning, anti-rollback protection, strict rule precedence,
exact URL ≠ domain approval, fail-safe on policy failure, minimal
sensitive logging, strong test coverage, tamper-aware cache, clear
blind spot documentation.

To move to "security-proven": external code review, penetration test,
formal test coverage report, incident response plan, bug bounty program,
compliance audit trail.

Product positioning: "LinkLook checks every web link that reaches it as
your default browser." Never claim device-wide or 100% protection.

SECTION 13 — SHORT CANONICAL SUMMARY
================================================================

The current canonical direction is:

1. LinkLook is a straightforward, security-driven browser for iOS focused on phishing
   and smishing prevention.
2. It is a security gateway, not a general-purpose browser product.
3. V1 is a browser-first release. Action extension is V2. Network Extension is V3.
4. The canonical outcome model is: allow (transparent) / inform / warn / block.
5. Privacy-first by default. Minimal data, no hidden profiling, transparent processing.
6. Two-data-flow separation: personal trust on-device, threat signals anonymous with
   differential privacy and k-anonymity (1,000+ threshold for global list).
7. Smart whitelist with risk-tiered re-validation, not a static allow list.
8. The core must remain independent from iOS/UI/Apple frameworks.
9. Behavior should be policy-driven where possible.
10. AI-ready from the start: structured interfaces, stable schemas, deterministic MVP
    base with clean insertion points for later AI.
11. AI comes before Network Extension in the roadmap. Core ML active-planned.
    Foundation Models optional. Custom SLM deferred. Claude backend not required.
12. Structured data should be captured so learning and future ML remain possible.
13. Development should stay test-first, simple, and disciplined.
14. The repo should have one source of truth for decisions — this log, governed by the
    canonical master decision log.
15. App Store rejection risk and browser entitlement risk are accepted showstopper risks.
16. User confusion, misuse risk, and redirect-button misinterpretation must be managed
    explicitly.
17. Proactive notification when previously cleared links are later reclassified as
    malicious — honesty principle in action.
18. Domain reputation database as a future data platform with public API and tiered
    B2B pricing.
19. Localization-ready from V1: reason codes language-neutral, UI uses Swift
    localized strings, Dutch + English from launch.
20. User-configurable browser preference: safe links open in the user's chosen
    browsing app. Safari is the default. LinkLook is never a redirect target.
    Preference stored on-device.
21. Preferences split: device-level settings (profile type, response mode) in
    iOS Settings bundle; user-experience settings (handoff browser, whitelist,
    notifications, donation tier) in the in-app settings screen. All preferences
    sync to shared App Group container for Network Extension access.
22. Recent checks history on by default, local only, clearable and disableable.
    Stores URL, verdict, reason code, timestamp — no page content. Supports
    Proof of Activity (PRD 5.5). Part of personal trust data (MDL-100).
23. Three-tier URL reuse model: exact normalized URL (strongest), URL pattern
    (medium, V2), domain reputation (weakest). Domain trust alone never approves
    a URL. Redirect chains tracked separately. Lookup order: exact URL → pattern
    → domain → redirect check → verdict.
24. Crowdsourced threat contribution requires explicit user opt-in (off by
    default). Anonymous domain-level signals only. Toggle in in-app settings.
    V2 backend; V1 must keep the two-data-flow separation clean.
25. Backend policy is a versioned JSON decision table with five sections:
    metadata, defaults, signal weights/thresholds, rules, and overrides.
    Code = engine, policy = behavior. Fetched on launch and foreground, cached
    locally with 7-day grace period. Resolves MDL-027.
26. Safe-link display preference: "always show result" (default, enforced for
    PROTECTED) vs "auto-open safe links" (optional for STANDARD). Only the
    allow path is affected. Core rule preserved — verdict computed first.
27. Entry path visibility: LinkLook receives URLs only via standard default-
    browser handoff. Universal links, custom schemes, App Clips, in-app
    browsers, and non-web content bypass LinkLook. Named limitations,
    documented honestly. V2 Action Extension and V3 Network Extension close
    progressively more gaps.
28. Enriched policy rule schema: per-rule validity windows, source provenance,
    override controls (allow_user_bypass, sticky), deterministic matching order
    (priority → specificity → severity), priority numbering guidance. An
    approved exact URL does not automatically approve the domain. No regex,
    no user-specific rules in V1. Messages excluded from policy — reason code
    mapping handles localization.
29. Policy spec split into two implementable docs: Policy_JSON_Schema_Draft.txt
    (data contract — JSON shape, field definitions, validation rules, V1 code
    mapping table, complete example policy) and Policy_Evaluation_Precedence.txt
    (runtime behavior — 8-step pipeline, matching order, conflict resolution,
    five fail-safe scenarios, determinism requirements, scope rules).
    Backend_Policy_Specification.txt remains the design rationale.
30. Security assessment: LinkLook is "security-promising, not yet security-
    proven." Five trust boundaries (iOS handoff, policy delivery, open-
    elsewhere, user bypass, on-device storage). Ten threats (policy backend
    compromise highest). Ten hardening requirements (HR-01–HR-10). To reach
    "security-proven": external review, pentest, formal coverage, incident
    response plan, bug bounty. Product positioning: never claim device-wide
    or 100% protection.
