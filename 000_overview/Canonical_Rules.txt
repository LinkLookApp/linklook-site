LinkLook — Canonical Rules Summary
Version: 1.1
Date: 2026-03-16

This file summarises the enforceable rules from CLAUDE.md (the single source of
truth in the LinkLook-App repository). It exists so that the LinkLook-Docs
package is self-contained — reviewers do not need access to the app repository
to understand the canonical model.

When this file and CLAUDE.md conflict, CLAUDE.md wins.

================================================================================
1. PRIMARY BROWSER MODEL
================================================================================

LinkLook is the user's primary browser. Users stay inside LinkLook for normal
browsing. There is NO user-facing "Open in other browser" option — not on
verdict screens, not in the toolbar, not anywhere.

Design target: familiar iPhone browser patterns. LinkLook should feel familiar
to Safari users while remaining clearly identifiable as LinkLook.

================================================================================
2. RISK-PROPORTIONAL SAFETY FLOW
================================================================================

Every entered or tapped URL is checked before any page loads.

Routing after the verdict:
- OK      → direct open in LinkLook + brief non-blocking toast
- INFORM  → interstitial with a forward path
- WARN    → stronger interstitial with extra-friction forward path
- BLOCK   → hard block, no forward path
- UNKNOWN → no forward path by default, retry allowed

The verdict is always computed before page rendering. Safe links do NOT show a
blocking interstitial.

================================================================================
3. NO-PRELOAD PRINCIPLE
================================================================================

LinkLook NEVER loads, fetches, or renders the destination page before the
relevant allowed state is reached.

- OK: allowed state is reached as soon as the verdict is computed, then the
  page opens immediately in LinkLook.
- INFORM / WARN: the page does not load until the user explicitly taps the
  forward action.
- BLOCK / UNKNOWN: no page load.

Before open, the safety UI may show only information derived from the URL or
from approved safety services: domain, full URL, verdict, reason, signal
badges, and approved Safe Preview output.

================================================================================
4. VERDICT UX
================================================================================

OK:
- No blocking screen
- Page opens immediately in LinkLook
- Brief non-blocking toast: "Looks OK — opening"

INFORM:
- One large "Go Back" button
- One medium "Check Message Context" button (Pro)
- One small "Continue →" button

WARN:
- One large "Go Back" button
- One medium "Check Message Context" button (Pro)
- One very small "Continue Anyway" button
- Extra confirmation tap before proceeding

BLOCK:
- One large "Go Back" button only
- No forward path

UNKNOWN:
- "Go Back" + retry option
- No silent continue

All forward paths open in LinkLook. There is NO option to open in another
browser.

================================================================================
5. SMART SYSTEM HANDOFF (EDGE CASES ONLY)
================================================================================

Some flows do not work well inside WKWebView: DigiD authentication, iDEAL
payment redirects, and OAuth flows. LinkLook detects these automatically by
matching known domains and hands off to the system. This is NOT a user-facing
button — it is automatic, invisible, and limited to a maintained allowlist.

================================================================================
6. BROWSER TOOLBAR (DURING BROWSING)
================================================================================

Bottom toolbar (Safari 5-button layout): Back, Forward, Share, Bookmarks, Tabs.
Share menu: Copy Link, Share….
Bookmarks: opens panel with Favorites + Reading List tabs.
Reload/Stop and Reader Mode live in the compact address bar.
No "Open in other browser" button anywhere in the toolbar.

================================================================================
7. CHECK PIPELINE
================================================================================

Run in parallel:
  (a) LinkLook structural analysis (always)
  (b) GSB Lookup API query (with API key)
  (c) On-device AI URL analysis (Pro only)
Combine results. Highest severity wins.
Severity order: OK < INFORM < WARN < BLOCK.

No safe-URL caching — a fresh check runs every time.

================================================================================
8. FREE / PRO TIERS
================================================================================

Phase 1 (Early Access): All features unlocked. Pro features show "Free during
Early Access" badge.

Phase 2 (Pro launch): Pro features gated behind StoreKit subscription. Free
users see Pro features as visible but locked.

Always Free: structural analysis, GSB checks, all verdict routing, all entry
points, full browser features, history.

Pro: context analysis, screenshot analysis, on-device AI URL analysis.
