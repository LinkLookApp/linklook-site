LinkLook Core User Flows
Version: v3.0
Date: 2026-03-10
Origin: ChatGPT, updated to align with Canonical Master Decision Log 2026-03-10

Canonical alignment note (2026-03-10)
LinkLook is a straightforward, security-driven browser for iOS. The canonical
outcome model is: allow (transparent) / inform / warn / block. Redirect is
part of the browser flow, not a separate product mode. The minimal-browser
fallback direction has been superseded (MDL-007).

Canonical UX flows

0. Start page — omnibox flow (app opened directly)
- the user opens LinkLook without tapping a link
- the screen is calm, clean, and almost empty
- one field does everything: URLs, domains, and search

Layout (top to bottom):
- omnibox: back arrow (when applicable), unified address/search field
  ("Enter website or search") — browser-style, establishes LinkLook as a
  real browser. No settings gear in the top bar (accessible via tab bar).
- app identity: "LinkLook" heading + tagline (see CLAUDE.md rule 17) —
  centered, only visible on start page
- tab bar: Home, Recent, Settings — persists across all screens
  NOTE: Recent sites are NOT shown on the Home/start page. History is only
  accessible from the Recent tab. The Home screen stays clean and focused.
  HOME BUTTON BEHAVIOR: Tapping Home ALWAYS navigates to the initial start
  page. It must dismiss any active WebView, cancel any in-progress check,
  and reset to the clean start page (omnibox + app identity). The Home
  button must never be a no-op — it always navigates, even if the user is
  already on the start page (in which case it clears the omnibox).

Omnibox input classification:
- Full URL (starts with http:// or https://) → use as-is
- Domain (contains dot, no spaces, e.g. bank.nl) → prepend https://
- Search text (everything else) → convert to Google search URL
All three cases go through the same safety check → result → browsing flow.

When browsing, a bottom toolbar appears with four buttons:
- Back, Forward, Reload/Stop, Open in preferred browser (shows browser icon)

Visual style: white background, generous spacing, rounded card elements,
light grey secondary text. No news, ads, bookmarks, or search suggestions.
No "paste from clipboard" button — users paste directly into the omnibox.

When iOS hands LinkLook a URL via default-browser handoff, the start page
is never shown — the check flow begins immediately.

Design principle: the start page should feel like a focused tool, not a
browser home page. Apple should see a genuine browser with a clear purpose.

Full visual spec: 140_ux/Start_Page_Design.txt
Visual reference: 140_ux/Start_Page_Mockup.html

0b. Checking progress flow (all links pass through this)
- after input classification, every link enters the checking state
- the UI shows a spinner with time-based status messages:
  0–2s "Checking link…", 2–5s "Still checking…", 5–8s "This is taking
  longer than usual…"
- no fake percentages, no technical jargon
- protection mode shown separately: "Basic protection" (Free) or
  "AI-enhanced protection" (Pro)
- if no result within 8 seconds → timeout screen with Retry / Go back /
  Open elsewhere (or Continue anyway depending on policy)
- never auto-mark a link safe on timeout, never spin indefinitely
- full spec: 140_ux/Checking_Progress_UX.txt

Result screen rules (apply to all verdict screens):
- Do NOT show a "Check another link" button. The Home tab in the tab bar
  already navigates back to the start page. Redundant navigation clutters
  the result screen. The result screen shows only: verdict, explanation,
  primary action, and secondary action if applicable.
- Button order: On all non-allow screens (inform, warn, block), "Go Back"
  is always the large primary button and "Go to Site" / "Open Anyway" is
  always the small secondary button. The safe choice is always prominent.

1. Allow flow (transparent)
- user taps a link
- LinkLook checks quickly (checking progress screen shown briefly)
- link is safe — user continues with minimal friction
- user barely notices LinkLook on the fast trusted path

2. Inform flow ("Heads Up")
- link has something worth noting but risk is low
- brief explanation shown
- primary action: "Go Back" (large button, return to start page)
- secondary action: "Go to Site" (small button, proceed to page)
- Go Back is always the prominent action on inform, warn, and block screens.
  Go to Site is always the smaller secondary option. This keeps the safe
  choice visually dominant across all non-allow verdict screens.

3. Warn flow (suspicious / uncertain)
- clear but calm warning
- explain uncertainty without panic
- safe action is primary
- continue anyway remains available if policy allows

4. Block flow (known phishing / malicious)
- clear warning with explanation
- safe action is primary (go back)
- blocked by default

5. Shortened link flow
- expand destination first
- show where the link really goes
- then evaluate through the normal outcome model

6. Network failure flow
- explain that the safety check could not complete
- fail informatively, not silently

7. False positive flow
- preserve trust by allowing a controlled override where appropriate
- "This doesn't look right" link on warn/block result screens
- swipe-to-report on any Recent tab row
- report sheet captures URL, verdict, reason, optional user comment
- full design in False_Positive_Recovery_Flow.txt

8. Education flow
- explain in plain language why a link is risky
- help users learn scam patterns over time

9. Redirect flow
- user chooses redirect option/button
- redirect sends the link to another browser
- wording and placement must make clear this does not mean LinkLook considers
  the destination safe (MDL-087)
- Safari redirect: iOS does not allow apps to force-open Safari when another
  browser is the system default. LinkLook uses SFSafariViewController (Safari's
  engine, presented in-app) to guarantee Safari. See CLAUDE.md rule 10.

10. Settings flow (MDL-104, MDL-105)

In-app settings (Settings tab in tab bar):
- user picks which browser safe links should open in (default: LinkLook).
  "LinkLook" is always the first option — it means "stay in LinkLook's
  built-in browser, no redirect." When LinkLook is selected, the redirect
  button is hidden from the browser toolbar entirely.
  The browser list MUST show all installed browsers on the device, not just
  Safari and Default. Use LSApplicationQueriesSchemes in Info.plist to declare
  supported URL schemes (googlechromes, firefox, brave, microsoft-edge, opera,
  ddgQuickLink, arc) and check availability at runtime with canOpenURL().
  Only show browsers that are actually installed. See CLAUDE.md rule 9.
- user picks safe-link display mode (MDL-109, "always show result" by default):
  — "always show result": safe result screen shown briefly for reassurance,
    then page loads in minimal browser
  — "auto-open safe links": safe links skip the result screen and load
    directly in the minimal browser
  — PROTECTED profile: always show result is enforced, cannot be changed
  — STANDARD profile: user can choose either mode
  — inform, warn, block verdicts always show their screens regardless
- user picks text size: Small, Medium (default), or Large. Affects the
  omnibox and result screens (titles, messages, buttons). Exact point sizes
  defined in CLAUDE.md rule 14. Stored on-device.
- user manages whitelist, notification preferences
- user can view, clear, or disable recent checks history (MDL-106, on by default)
- user can enable crowdsourced threat contribution (MDL-108, off by default, V2)
  — plain-language label like "Help improve safety for everyone"
  — when enabled, anonymous domain-level signals may be submitted
  — toggle is off by default because data leaves the device
- foundation donation tier shown read-only
- LinkLook is always the first browser option (means "stay in built-in browser")
- preferences stored on-device, respected by all redirect paths
  (result screen, in-app browser toolbar)
- if the chosen browser is later removed, LinkLook falls back to Safari

11. History flow (MDL-106)
- every link check is recorded locally: URL, verdict, reason code, timestamp
- user can view recent checks from the Recent tab (NOT the Home screen)
- every entry in the Recent list MUST be tappable: tapping re-runs the check
  (for URLs) or re-opens the search (for search queries)
- history supports Proof of Activity (PRD 5.5) — user sees what LinkLook has
  done for them
- user can clear all history from in-app settings
- user can disable history recording from in-app settings
- no page content, screenshots, or cookies are stored
- history data never leaves the device (personal trust data, MDL-100)

iOS Settings bundle (device-level):
- profile type (PROTECTED / STANDARD)
- response mode (block / warn)
- set during setup or by an adult child installer

Acceptance-test note
Each canonical user flow should map to one or more automated acceptance tests
so UX behavior is stable across releases.

Additional UX test note
The accidental-click reassurance case should be treated as an explicit UX
scenario. When a user taps a link impulsively or by mistake, LinkLook should
quickly show that a safety check is in progress so the user immediately feels
protected rather than lost. Full design: 140_ux/Checking_Progress_UX.txt §7.
