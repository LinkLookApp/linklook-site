================================================================================
PURPOSE
================================================================================

This document describes the start page — the screen the user sees when they
open LinkLook directly (not via a link handoff). It is the home state of the
app.

When iOS hands LinkLook a URL via default-browser handoff, this screen is
never shown. The check flow begins immediately.

2. ELEMENT DESCRIPTIONS
================================================================================

  2.1 OMNIBOX (TOP BAR)
  ──────────────────────
  A unified address/search bar that establishes LinkLook as a real browser
  for Apple App Review. This is the single entry point for all navigation.

  Elements:
  - Back chevron (←) — a plain chevron.left icon, shown when not in browsing
    mode; greyed out on the start page since there is no browsing history yet.
    During browsing, back/forward move to the bottom browser toolbar. Its size
    scales with the user's text size setting (prefStore.fontSize.backButtonSize):
    Small = 16pt, Medium = 18pt, Large = 22pt. Consistent with standard
    browser conventions — no text label, just the chevron.
  - Omnibox field — placeholder text: "Search or link"
    Keyboard type: .webSearch (gives the user a space bar for search text).
    Font size: large enough for elderly users to read comfortably without
    glasses. Exact minimum defined in CLAUDE.md (rule 14). The field must
    also support Dynamic Type scaling.
    Accepts URLs, domain names, and search queries (see §2.1a below).
  - Go button (→) — a green circular button at the right end of the
    omnibox field. Tapping it submits the current omnibox input (same as
    pressing Return on the keyboard). Visible whenever the omnibox has
    text. Hidden when the omnibox is empty (placeholder state).
    Exact styling (SF Symbol, color, size) defined in CLAUDE.md (rule 15).
    Provides an explicit tap target for users who may not instinctively
    press the keyboard Return key.
  - No settings gear — settings are accessible via the tab bar only.

  The top bar persists across all screens. When browsing, it shows the
  current URL. On the start page, it shows the placeholder.

  2.1a INPUT CLASSIFICATION (omnibox logic)
  ──────────────────────────────────────────
  The omnibox classifies user input into three cases:

  A. Full URL — starts with http:// or https://
     Examples: https://bank.nl/login, http://example.com
     Action: use as-is, run safety check

  B. Domain / host-like — contains a dot (.) and no spaces
     Examples: bank.nl, www.amazon.com, 192.168.1.1
     Action: prepend https://, run safety check

  C. Search text — everything else (contains spaces, or no dot)
     Examples: best noise cancelling headphones, what is smishing
     Action: convert to https://www.google.com/search?q=<encoded>,
             run safety check on the Google URL

  Fixed search engine for v1: Google. No search-engine settings.

  After classification and normalization:
  - URL and domain inputs go through the safety check → verdict → browsing flow.
  - Search queries skip the safety check entirely and open Google results
    directly in LinkLook's built-in browser. No verdict screen (CLAUDE.md rule 7).

  2.1b "OR" SEPARATOR + QR BUTTON
  ─────────────────────────────────
  Directly below the omnibox, a small green "or" label
  (prefStore.fontSize.orLabelSize: Small 14pt, Medium 16pt, Large 19pt)
  connects the omnibox to the QR code scanner button below it. This makes
  the two entry points (type/paste vs scan) visually connected.

  The QR button is a green capsule: icon (24pt) + "Scan QR code" (18pt).

  2.2 APP IDENTITY
  ─────────────────
  Vertically centered in the remaining space below the QR button (using
  GeometryReader, minHeight = 45% of visible area). Two elements:
  - LinkLook logo — the SVG "eyes" version (LINK on top, LOOK below with
    chain-link eyes on the OO letters). Rendered from the LinkLookLogo
    image asset (separate from AppIcon). Creamy white background (#FAF8F4)
    matching Theme.creamyWhite. No tagline in the SVG.
    Displayed at 220×150pt with .renderingMode(.original).
  - Tagline — "A safety-driven browser" (CLAUDE.md rule 18), rendered as
    separate SwiftUI text in Theme.logoGreen (#2E7D4F). Scales with
    text size setting (prefStore.fontSize.taglineSize: Small 18pt,
    Medium 20pt, Large 24pt).

  This section is only visible on the start page. It disappears when
  browsing begins.

  2.3 VERDICT STATS CARD
  ───────────────────────
  Below the app identity, a compact card shows verdict statistics when
  the user has history entries. It provides "proof of work" — showing
  that LinkLook is actively protecting the user.

  Content:
  - Header: total links checked (all-time)
  - Verdict pills: OK / Info / Warn / Block counts (color-coded)
  - Weekly breakdown: last 4 weeks with mini verdict bars (only weeks
    with activity are shown)

  Rules:
  - Only visible when history is enabled and at least 1 entry exists
  - Not tappable — informational only
  - Disappears when browsing begins (same as app identity)
  - If history is cleared or disabled, the card disappears

  2.4 RECENT SITES — REMOVED FROM HOME SCREEN
  ──────────────────────────────────────────────
  Recent sites are NOT shown on the Home/start page. History is only
  accessible from the Recent tab in the tab bar.

  In the Recent tab, each row shows:
  - Domain name (left-aligned, bold)
  - Status label (right-aligned, lighter text):
    "Previously checked" — the link was checked and a verdict was reached
    "Not opened yet" — the link was checked but the user did not proceed

  Every entry in the Recent list MUST be tappable. Tapping re-runs the
  check (for URLs) or re-opens the search (for search queries).

  If history is empty, the Recent tab shows an empty state message.
  If the user has disabled history (MDL-106), the Recent tab indicates
  that history recording is turned off.

  2.5 TAB BAR
  ────────────
  A standard iOS tab bar at the bottom with three items:
  - Home — ALWAYS returns to the initial start page. Tapping Home must:
    (1) dismiss any active WebView / browsing session
    (2) cancel any in-progress link check
    (3) reset the screen to the clean start page (omnibox + app identity)
    The Home button must never be a no-op. It always navigates.
  - Recent — opens the full history screen
  - Settings — opens in-app settings. The Settings tab icon is always
    the standard gear icon — never changes. See CLAUDE.md rule 19.

  The tab bar persists across all screens.

  2.6 BROWSER TOOLBAR (bottom, during browsing)
  ───────────────────────────────────────────────
  When the user is browsing (WebView active), a bottom toolbar appears
  with four buttons, evenly spaced:

  - Back (chevron.left) — go back in web history, or exit browsing
  - Forward (chevron.right) — go forward in web history
  - Reload/Stop (arrow.clockwise / xmark) — reload page or stop loading
  - Share (square.and.arrow.up) — opens share menu with: Copy Link,
    Share…, Add to Favorites. No "Open in other browser" option.
  - Tabs (square.on.square) — opens tab overview grid (Safari-style).
    See CLAUDE.md rules 19, 49 for the full spec.
  NOTE: There is NO "Open in other browser" button. LinkLook IS the
  browser. Users stay in LinkLook at all times. See CLAUDE.md rule 10.

  Keyboard behavior: when the keyboard appears (user taps the omnibox
  to type a new URL while browsing), the browser toolbar shifts up to
  sit above the keyboard. The WebView shrinks to accommodate. When the
  keyboard dismisses, the toolbar returns to its original position.
  This ensures the toolbar is always accessible.

3. VISUAL STYLE
================================================================================

  Brand colors:
  - Primary green:    #4CAF76  (medium green — buttons, active tab, accents)
  - Logo green:       #2E7D4F  (dark green — "LinkLook" wordmark only)
  - Green dark:       #3D8F62  (hover/pressed state)
  - Green light:      #E8F5ED  (subtle green tint for top bar background)
  - Creamy white:     #FAF8F4  (main background — warm, not clinical white)
  - Card surface:     #F2EFE9  (slightly darker cream for card backgrounds)
  - Item surface:     #FDFCFA  (near-white for list items inside cards)
  - Text dark:        #2C2C2C  (headings, primary text)
  - Text medium:      #6B6B6B  (body, secondary text)
  - Text light:       #9E9E9E  (placeholders, status labels)
  - Border/divider:   #E5E0D8  (warm divider, tab bar separator)

  Application:
  - Background: creamy white (#FAF8F4), not pure white
  - Cards: slightly darker cream (#F2EFE9), rounded corners (iOS style)
  - Primary button: medium green (#4CAF76), white text
  - Top bar: light green tint (#E8F5ED) with subtle green border
  - Active tab: green text (#4CAF76), inactive tabs grey (#9E9E9E)
  - Navigation arrows and gear icon: green (#4CAF76)
  - Text hierarchy: large bold for headlines, regular weight for body,
    lighter grey for secondary text
  - Spacing: generous white space between elements
  - No decorative images, no gradients, no complex visual effects
  - No news, ads, trending content, bookmarks grid, or search suggestions
  - The overall impression: calm, warm, professional, trustworthy

  All icons use SF Symbols (Apple system icon library, available from iOS 13+).
  Rendering mode: monochrome. Weight: regular (matching body text weight).
  SF Symbols scale automatically with Dynamic Type.

  Disabled state for nav arrows:
  - On the start page, both arrows are disabled (no browsing history).
  - Disabled = same color (#2E7D4F) at 35% opacity.
  - Enabled = full color, tappable.

  VoiceOver labels:
  - Back arrow:            "Back"
  - Omnibox field:         "Enter website or search"
  - Go button:             "Go"
  - Home tab:              "Home"
  - Recent tab:            "Recent sites"
  - Settings tab:          "Settings"
  - Recent site row:       "{domain}, {status}" (e.g. "bank.nl, Previously checked")

  - Toolbar: reload:       "Reload"
  - Toolbar: stop:         "Stop loading"

  Accessibility:
  - All text must pass WCAG AA contrast ratio
  - Button targets must meet minimum 44pt touch target (Apple HIG)
  - VoiceOver labels on all interactive elements (see above)
  - Dynamic Type support for all text and SF Symbols

================================================================================
4. STATES
================================================================================

  4.1 FIRST LAUNCH (empty state)
  - Start page shows only: omnibox, app identity, tab bar
  - Clean and minimal — just the field and the app name

  4.1b NOT-DEFAULT STATE (after onboarding, no handoff received yet)
  - A gentle reminder card appears below the app identity, prompting the
    user to set LinkLook as default browser.
    See Default_Browser_Activation_Flow.txt section 5.
  - Card disappears once LinkLook receives its first URL handoff.

  4.2 RETURNING USER (normal state)
  - Same as first launch: omnibox, app identity, tab bar
  - No recent sites on the Home screen (history is in the Recent tab)

  4.3 HISTORY DISABLED
  - Home screen unchanged (history was never shown here)
  - Recent tab shows "History recording is off" message

  4.4 CHECKING (link check in progress)
  - Spinner + time-based status messages shown in content area
  - Full spec: Checking_Progress_UX.txt
  - Go button hidden, omnibox shows the URL being checked
  - Back arrow enabled (cancels check, returns to start page)

  4.5 BROWSING (WebView active)
  - Omnibox shows current URL (or search query text if search-originated)
  - Page content fills the area between omnibox and browser toolbar
  - Bottom browser toolbar visible: back, forward, reload/stop, share, tabs

================================================================================
5. DESIGN PRINCIPLES
================================================================================

  5.1 One thing well
  The start page communicates: "this app does one thing — check links."

  5.2 Calm confidence
  The user should feel safe and in control. No urgency, no alarm, no
  marketing pressure.

  5.3 Real browser, focused purpose
  Apple must see a genuine browser with navigation, address bar, and
  browsing capability. But the purpose is clear: safety first.

  5.4 Progressive disclosure
  First launch is minimal. Returning users see recent sites. Power users
  find settings in the tab bar. Nothing is forced.
