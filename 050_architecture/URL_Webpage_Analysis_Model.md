# URL & Webpage Analysis Model

> **Status:** Draft — v2
> **Date:** 2026-03-17
> **Scope:** Defines LinkLook's two-step analysis architecture: URL check first,
> webpage check only after explicit user action. Covers the four technical
> artifacts, privacy-reduced analysis, optional deeper analysis, audit logging,
> technical non-logging controls, V1 build scope, and testing requirements.
> **Enforceable rules:** `CLAUDE.md` (rules 104–117).
> **Related specs:** `URL_Privacy_Signal_Spec.md`, `Cloud_Analysis_Consent_Spec.md`,
> `Safety_Flow_UX.md`, `Safe_Preview.md`.

---

## 1. Executive Summary

LinkLook uses a **two-step trust model**:

1. **Step 1 — URL check.** Evaluate the URL and give a verdict.
2. **Step 2 — Webpage check.** Only after explicit user action, fetch the page
   and perform deeper analysis.

Within webpage analysis, the default is **privacy-reduced analysis**. Full-page
analysis is an explicit deeper option, not the normal path. When deeper analysis
is selected, LinkLook keeps a minimal audit log showing the user actively chose
that mode.

**One-sentence summary:** LinkLook separates URL and webpage checking, uses
privacy-reduced webpage analysis by default, allows deeper full-page analysis
only by explicit user choice, logs that choice minimally, and prevents raw
full URL/page content from being logged by design.

### Why this is the right V1 approach

This model works because it is:

- **Simple** — avoids too many branches and advanced consent mechanisms.
- **Privacy-aware** — separates lighter URL processing from deeper webpage processing.
- **User-friendly** — the user is not asked to make technical choices too early.
- **Defensible** — you can clearly explain what was checked, when the page was
  fetched, what type of analysis was used, and what was and was not logged.
- **Buildable** — realistic for V1 without becoming overly complex.

---

## 2. Core Product Rule

> URL checking and webpage checking are not the same thing. Therefore they
> should be treated as two separate steps.

This separation improves privacy (the page is never fetched unless the user
asks), clarity (the user understands what each step does), explainability
(each verdict has a clear basis), user trust (no hidden escalation), and
auditability (each step can be independently logged and reviewed).

## 3. The Four Technical Artifacts

LinkLook's analysis pipeline produces four distinct artifacts. Each has a
specific purpose, audience, and privacy posture.

### 3.1 Original URL

The full link as entered, pasted, or tapped by the user.

**Use only where necessary:**
- Actual page fetch (when user requests webpage check)
- Actual navigation (when user taps "Continue"/"Continue Anyway")
- Backend page retrieval

**Avoid showing by default** when it contains sensitive data (use the stripped
URL instead). Never log the original URL before the privacy scan completes
(`CLAUDE.md` rule 101).

### 3.2 Stripped URL

A privacy-reduced version of the original URL, produced by the on-device
`URLPrivacyScanner` and `URLPrivacyMasker` (spec: `URL_Privacy_Signal_Spec.md`).

**Use for:**
- User display (on verdict screens, in history)
- Explainability (showing which parts triggered signals)
- Privacy signaling (user sees that sensitive data was hidden)
- Low-risk backend analysis (`/analyze-url` with masked URL)
- Logging references (hash or reduced form only)

**Key principle:** Hide sensitive values, but preserve risk-relevant structure.

What to keep:
- Domain (always)
- Path structure (segment count, depth)
- Query parameter names (when useful for signal detection)
- Redirect targets (if relevant)
- TLD, subdomain structure

What to hide:
- Email addresses, tokens, IDs, account values, session values
- Personal names, phone numbers, addresses
- Health-related terms in parameter values

### 3.3 Preview Page

A tightly limited visual representation of the webpage for the user. This is
the Safe Preview feature (spec: `Safe_Preview.md`).

**Examples:**
- Static JPEG screenshot (current implementation via Urlbox)
- Grey/desaturated preview (WARN verdict)
- Full-color preview (INFORM verdict)

**Purpose:**
- Help the user recognize the page visually
- Reduce blind trust ("does this look like my bank?")
- Keep preview separate from actual navigation

**This is for the human.** It is visual context, not analysis input.

### 3.4 Stripped Webpage Representation

A reduced page model generated by the backend for AI analysis. This is the
privacy-reduced version used by default in webpage analysis.

**What to preserve (security-relevant signals):**
- Page title
- Visible headings (h1–h3)
- Visible text content (main body)
- Call-to-action labels (buttons, links)
- Form labels (login, payment, download, contact)
- Presence of login/payment/download/remote-access prompts
- Visible brand names and logos (text references)
- Outbound domains (links pointing away from the current domain)
- Urgency language indicators
- Mismatch signals (domain vs brand, URL vs page content)

**What to reduce or omit:**
- Raw personal data (autofilled values, user-specific content)
- Unnecessary identifiers (session tokens, tracking parameters)
- Tracking/session data (cookies, analytics scripts)
- Raw technical clutter (CSS, inline scripts, metadata not needed for scoring)
- Full page HTML (use extracted text, not raw source)

**This is for the machine.** It provides the AI with enough signal for
fraud/risk assessment while minimizing privacy exposure.

## 4. Two-Step User Flow

### Phase 1 — URL Phase

```
User taps / pastes / types URL
        │
        ▼
Client normalizes the URL
        │
        ▼
Client checks for privacy-sensitive data (URLPrivacyScanner)
        │
        ▼
Client generates stripped URL (URLPrivacyMasker)
        │
        ▼
On-device checks: structural analysis + GSB + on-device AI (Pro)
        │
        ▼
Cloud URL analysis if enabled (sends stripped or full URL per user choice)
        │
        ▼
URL verdict rendered
        │
        ▼
User sees:
  • URL verdict (OK / INFORM / WARN / BLOCK)
  • Stripped URL (sensitive data hidden)
  • Privacy notice (if data was hidden)
  • Signal badges + reasons
  • "Check webpage with AI" button (if INFORM or WARN)
        │
        ▼
User chooses:
  • Go Back
  • Continue / Continue Anyway (open page in LinkLook)
  • Check webpage with AI → enters Phase 2
```

### Phase 2 — Webpage Phase

```
User taps "Check webpage with AI"
        │
        ▼
Backend fetches the page (using original or masked URL per privacy choice)
        │
        ▼
Backend generates:
  • Preview page (JPEG screenshot via Urlbox)
  • Stripped webpage representation (extracted text, headings, forms, CTAs)
        │
        ▼
AI analyzes stripped webpage representation (privacy-reduced by default)
        │
        ▼
Webpage verdict rendered
        │
        ▼
User sees:
  • Webpage verdict
  • Preview page (visual screenshot)
  • Reasons (what the AI found)
  • Confidence indicator
        │
        ▼
If confidence is low:
  "Result uncertain — Run deeper analysis? This uses more page content."
        │
        ▼
User chooses:
  • Go Back
  • Continue (open page in LinkLook)
  • Run deeper analysis → full-page analysis (audit logged)
```

## 5. Privacy-Reduced vs Full-Page Analysis

### Default: Privacy-reduced analysis

The AI analyzes the **stripped webpage representation** — enough signal for
most fraud/risk detection, with minimal privacy exposure.

Most scam signals are visible in the stripped representation: fake login forms,
brand impersonation, urgency language, suspicious CTAs, payment requests. The
stripped version captures these because they are part of the page's visible
content structure.

### Optional: Full-page analysis

The AI analyzes **more page content** — raw body text, additional metadata,
full link graph, deeper structural analysis. This mode is used only when:

- The user explicitly taps "Run deeper analysis"
- The confidence from privacy-reduced analysis is low
- The stripped representation likely missed relevant context

### Can stripped analysis produce a different result than full?

**Yes.** Stripping removes information, which can create two effects:

1. **Stripped appears safer than full (false-negative risk).** Important
   suspicious details were in the removed data. Example: full version is WARN,
   stripped version is INFORM or OK.

2. **Stripped appears more suspicious than full (false-positive risk).**
   Legitimate context was in the removed data. Example: full version is OK,
   stripped version is INFORM or WARN.

**Design principle:** LinkLook does not promise that stripped analysis is
identical to full analysis. Instead:
- Preserve risk-relevant structure in the stripped version
- Preserve meaningful visible signals
- Escalate to INFORM when confidence drops because context was removed

## 6. Confidence-Based Escalation

The strongest UX model for deeper analysis:

1. User taps "Check webpage with AI"
2. LinkLook performs privacy-reduced analysis
3. If the AI's confidence is high → show the result directly
4. If confidence is low → offer escalation:

```
┌──────────────────────────────────────┐
│                                      │
│  Result uncertain                    │
│                                      │
│  The privacy-reduced check could     │
│  not reach a confident conclusion.   │
│                                      │
│  Run deeper analysis?                │
│  This uses more page content.        │
│                                      │
│  ┌──────────────────────────────────┐│
│  │  Run deeper analysis             ││
│  └──────────────────────────────────┘│
│  ┌──────────────────────────────────┐│
│  │  Keep current result             ││
│  └──────────────────────────────────┘│
│                                      │
└──────────────────────────────────────┘
```

This gives:
- Privacy-first behavior by default
- More accuracy when actually needed
- Less unnecessary user burden

## 8. Audit Logging

### Why logging matters

If LinkLook offers full-page analysis as an option and uses the user's choice
as the justification for processing more page data, it should keep a minimal
audit record showing the user actively selected that mode. This demonstrates:
- The user actively chose deeper analysis
- Deeper processing was not automatic or hidden
- The behavior matched the UI shown to the user
- The product follows a privacy-aware approach

### What to log

**Recommended minimal audit fields:**
- Pseudonymous user or device ID
- Timestamp
- Event type (e.g. `full_page_analysis_selected`, `reduced_analysis_completed`)
- Request ID
- URL hash or reduced URL reference (never the raw URL)
- Consent/UI text version shown to the user
- Analysis mode chosen: `reduced` or `full`
- Purpose (e.g. `webpage_fraud_analysis`)

**Optional additional fields:**
- App version
- Policy version
- Whether the choice was user-initiated or system-escalated

### What NOT to log

- The raw full URL
- Full page contents
- Screenshots in the audit log
- The full AI analysis payload
- Raw extracted sensitive values

If any of these must be stored for debugging or abuse analysis, they require
separate justification, tight access controls, and retention limits.

### Retention

Keep audit logs only as long as needed for accountability, debugging, abuse
review, and compliance. Avoid indefinite retention.

**Practical rule:** Keep minimal audit events for a limited period (90 days).
Avoid retaining content-level data unless separately justified.

## 9. V1 Recommendation

### V1 choice model

V1 exposes one primary button ("Check webpage with AI") that runs
privacy-reduced analysis by default. Deeper full-page analysis is available
as a secondary action:

- **Primary action:** "Check webpage with AI" → runs reduced analysis
- **Secondary action (optional, advanced):** "Use deeper full-page analysis"
  → runs full analysis, audit logged

The user is never forced to choose between modes upfront. Reduced analysis
is always the default path.

See `CLAUDE.md` rules 108–110 for enforceable rules.

### V1.1 enhancement: confidence-based escalation

A stronger model for a post-launch update:

1. Run reduced analysis first
2. If confidence is low, show: "Result uncertain — Run deeper analysis?
   This uses more page content."

For V1, the simpler secondary-action approach is sufficient. The confidence-
based escalation can be added when confidence scoring is calibrated from
real-world data.

### V1 policy line

> LinkLook uses privacy-reduced webpage analysis by default. Full-page analysis
> is an explicit deeper option and, when selected, the user's choice is recorded
> in a minimal audit log.

### V1 proof model

For V1, the best proof that full analysis was used on purpose is not a
cryptographic mechanism. It is a combination of:

- Explicit separate user action (tap "Use deeper full-page analysis")
- Clear explanatory text (versioned UI copy shown to the user)
- Minimal audit record (log the decision, not the content)
- Technical controls preventing raw-content logging (see §9a)
- Automated tests proving logs stay clean (see §9b)

This is strong enough for V1 and much easier to implement correctly.

**Device-signed or cryptographic user-proof mechanisms** (e.g. Secure
Enclave signatures, passkey-style consent receipts) are a good future
enhancement but should not be part of V1. They add engineering complexity,
testing complexity, and failure modes that are not needed for a solid initial
privacy/accountability position.

## 9a. Technical Non-Logging Controls (V1)

> Raw full URL/page logs should not be generated by default. This should be
> solved mainly by architecture and controls, not by promise.

### Recommended V1 controls

1. **Structured allowlisted logging only** — Application logs use a defined
   schema. Only explicitly listed fields are written. No free-text request
   or response dumping.
2. **No request-body logging** — HTTP middleware and proxy layers must not
   log request bodies by default. Request bodies may contain the original URL
   or page content.
3. **Query string sanitization** — Before any log write, strip or mask query
   parameters that match sensitive key patterns. Use the same `URLPrivacyScanner`
   rules for consistency.
4. **Page content never in application logs** — Page text, HTML, and screenshots
   are processed in memory, passed to the AI, and discarded. They are never
   written to normal application logs.
5. **Crash/reporting tools configured** — Sentry, Crashlytics, or equivalent
   must be configured to exclude sensitive payload fields from crash reports.
   Strip URL query strings, request bodies, and response bodies from automatic
   breadcrumbs.
6. **Infrastructure/proxy logs reviewed** — Fly.io access logs, nginx/caddy
   logs, and any reverse proxy logs must be reviewed for URL leakage. Configure
   to log only method + path (no query string) or use hashed references.
7. **Automated canary tests** — CI tests that inject fake sensitive markers
   (e.g. `canary_email=test@canary.linklook`, `canary_token=CANARY12345`) into
   test URLs, run the full pipeline, then grep all log outputs for the canary
   values. If any canary value appears in any log, the test fails. This is the
   strongest V1 guarantee that sensitive data does not leak into logs.

### Why controls beat promises

Telling users "we don't log your data" is a promise. Technical controls that
prevent logging by design are proof. For V1, the combination of structured
logging, sanitization, and canary tests is much stronger than a policy
document alone.

## 9b. V1 Testing Requirements

### Mandatory V1 tests

1. **Webpage fetch requires explicit action** — Test that the backend does
   NOT fetch the page until the user taps "Check webpage with AI." Verify no
   HTTP request to the destination occurs during the URL phase.
2. **Reduced analysis is the default** — Test that `/analyze-page` uses
   `mode=reduced` when called without an explicit mode parameter. Verify the
   AI receives a stripped representation, not full page content.
3. **Full analysis requires explicit user action** — Test that `mode=full`
   is only sent when the user explicitly taps "Use deeper full-page analysis."
   Verify the backend rejects auto-escalation from reduced to full.
4. **Raw URL/page content not in logs** — Canary tests (see §9a.7). Inject
   known marker values, run the pipeline, verify markers do not appear in any
   log output (application logs, audit logs, crash reports, proxy logs).
5. **Audit records contain only minimal fields** — Test that audit log entries
   for `full_page_analysis_selected` contain only the fields listed in §8.
   Verify no raw URL, page content, or screenshot data in audit entries.
6. **Stripped URL preserves structure** — Test that the stripped URL retains
   domain, path structure, and query parameter names while hiding sensitive
   values. Round-trip URL validity check.
7. **Stripped webpage preserves signals** — Test that the stripped webpage
   representation contains risk-relevant fields (title, headings, form labels,
   CTAs, brand references) and omits raw personal data and tracking clutter.

### Test fixture approach

Use clearly synthetic test URLs with embedded canary markers:

```
https://my-test-bank-2026.xyz/login?email=canary@test.linklook&token=CANARY_TOKEN_12345&uid=CANARY_UID_99999
```

These URLs must NOT resolve to real sites (per `CLAUDE.md` rule 3).

## 9c. What to Build in V1

### Client side

- URL intake and normalization
- Sensitive URL detection (`URLPrivacyScanner`)
- Stripped URL generation (`URLPrivacyMasker`)
- URL verdict screen with stripped URL display
- Privacy notice when URL data is hidden
- Separate "Check webpage with AI" action (INFORM/WARN only)
- Optional "Use deeper full-page analysis" secondary action
- Clear explanatory text for each action

### Backend

- Fetch page only after explicit user action (never during URL phase)
- Create static preview (JPEG screenshot via Urlbox)
- Create stripped webpage representation for AI
- Run AI on stripped representation by default (`mode=reduced`)
- Run full analysis only when explicitly requested (`mode=full`)
- Store minimal audit event for full analysis choice
- Prevent raw URL/page content from entering logs (§9a controls)
- `/analyze-url` must remain URL-only (never fetch page)

### Testing

- All seven mandatory tests from §9b
- Canary tests in CI pipeline
- Unit tests for scanner, masker, and stripping logic
- Integration tests for the full two-step flow

## 10. Backend Implementation

### Stripped webpage generation

The backend generates the stripped webpage representation using:

1. **Page fetch** — Urlbox or trafilatura for text extraction
2. **Content extraction** — title, headings, visible text, form labels, CTAs,
   outbound domains, brand references
3. **Privacy reduction** — remove autofilled values, session data, tracking
   parameters, unnecessary identifiers
4. **Signal preservation** — keep urgency language, mismatch indicators,
   login/payment/download prompts

### Endpoint mapping

| Step | Endpoint | What it does |
|------|----------|-------------|
| URL check | `/analyze-url` | Analyze URL string only. Never fetch page. |
| Page preview | `/fetch-preview` | Fetch page, return JPEG screenshot. |
| Reduced webpage analysis | `/analyze-page` with `mode=reduced` | Fetch page, extract text, AI analyzes stripped representation. |
| Full webpage analysis | `/analyze-page` with `mode=full` | Fetch page, AI analyzes full content. Audit logged. |

The `mode` parameter on `/analyze-page` controls the analysis depth. Default
is `reduced`. The `full` mode requires explicit user action and generates an
audit log entry.

### What to send the AI (reduced mode)

```json
{
  "url_hash": "sha256:...",
  "domain": "example.com",
  "page_title": "Login - Example Bank",
  "headings": ["Welcome back", "Sign in to your account"],
  "visible_text_excerpt": "Enter your email and password...",
  "cta_labels": ["Sign In", "Forgot Password", "Create Account"],
  "form_labels": ["Email", "Password"],
  "has_login_form": true,
  "has_payment_form": false,
  "has_download_prompt": false,
  "outbound_domains": ["cdn.example.com", "analytics.example.com"],
  "brand_references": ["Example Bank"],
  "urgency_indicators": [],
  "domain_brand_mismatch": false
}
```

### What to send the AI (full mode)

The full extracted page text, link graph, metadata, and structural analysis.
Same JSON shape but with additional fields and untruncated text content.

## 11. Interaction with Existing Specs

### URL Privacy Signal (`URL_Privacy_Signal_Spec.md`)

The stripped URL (artifact 3.2) is produced by the same `URLPrivacyScanner`
and `URLPrivacyMasker` described in the Privacy Signal spec. The Privacy
Signal dialog controls whether the stripped or original URL is sent to the
backend. This model extends that concept to the webpage phase: the stripped
webpage representation is the page-level equivalent of the stripped URL.

### Cloud Analysis Consent (`Cloud_Analysis_Consent_Spec.md`)

The consent toggles control whether each step happens. The two-step model adds
a new consideration: webpage analysis (`/analyze-page`) now has two modes
(`reduced` and `full`). The consent toggle for Deep Page Analysis permits
both modes. The choice between reduced and full is a separate user action
within the webpage phase.

### Safe Preview (`Safe_Preview.md`)

The preview page (artifact 3.3) is the Safe Preview feature. No change to
the Safe Preview architecture. The preview is generated alongside the stripped
webpage representation when the user enters Phase 2.

## 12. Design Principles

1. **URL checking and webpage checking are separate steps.** Never silently
   escalate from URL check to page fetch.
2. **Default to privacy-reduced analysis.** The standard path uses the least
   data needed for an effective check.
3. **Full-page analysis is explicit.** The user must actively choose it.
4. **Log the decision, not the content.** Audit logs record what the user
   chose, not what the page contained.
5. **Confidence-based escalation.** Offer deeper analysis only when reduced
   analysis is genuinely uncertain.
6. **Do not promise equivalence.** Stripped analysis may produce different
   results than full analysis. That is acceptable — it is a trade-off between
   privacy and completeness.
7. **Preserve signal, reduce exposure.** Stripping should be privacy-reducing
   AND signal-preserving, not privacy-maximizing at the cost of detection.

---

## 13. What NOT to Do

- Silently run full-page analysis by default
- Automatically fetch the page for AI before user approval
- Force the user to choose "full or reduced?" every time
- Use "stripped", "full", or "reduced DOM" as raw technical labels in the UX
- Log full URLs or full page contents just to prove consent
- Promise that stripped analysis is identical to full analysis
- Add cryptographic proof mechanisms before the basic architecture is correct
- Rely only on policy for non-logging — use technical controls (§9a)
- Allow request-body logging in middleware or proxies
- Allow crash/reporting tools to capture sensitive payloads by default
