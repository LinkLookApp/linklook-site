ARCHITECTURE CONSTITUTION
================================================================================

CORE PRINCIPLE
Security > Convenience > Performance.

FLOW
Input → Normalize → Check → Decide → Explain → Act.

LAYERED DEFENSE
Deterministic → Structural Analysis → GSB → Optional AI Signals → Combiner.

PRIVACY BY DESIGN
No surveillance, minimal data retention. AI runs on-device only.

FAIL-SAFE
Warn on uncertainty, never silent allow.

================================================================================
SYSTEM OVERVIEW (V1)
================================================================================

LinkLook is a safety-driven browser for iOS. The v1 architecture is
client-side with an optional backend reputation service.

The check pipeline runs entirely on-device for the base product.
No external LLM is required at any stage.

ON-DEVICE RESPONSIBILITIES (v1)
- Capture user input (omnibox, QR, clipboard, Share Sheet)
- Normalize and classify input (URL vs search query)
- Run the full check pipeline (see below)
- Display verdict screen with appropriate buttons
- Let the user choose: go back, continue in LinkLook, or open in browser
- Maintain history (Recent screen)

OPTIONAL BACKEND
- Reputation lookup (additional signal, not required)
- If backend is unavailable, the on-device pipeline handles the check alone
- Backend results merge with local results (highest severity wins)

PIPELINE ORDER SUMMARY:
  Input capture → Normalization → Policy + deterministic checks →
  Static signal extraction → Decision engine → Optional AI URL signal →
  Deterministic combiner → UX mapping → Verdict screen.

IMPORTANT NOTE:
Backend LLM analysis is not a required runtime stage in the current roadmap.
If introduced later, it must enter through the signal interface and remain
non-authoritative (see AI_Boundary_Rules.txt rule 2).

================================================================================
SIGNAL MERGE RULES
================================================================================

The combiner is deterministic, not ML. It merges signals from all sources.

Severity order: allow < inform < warn < block.

GSB merge (CLAUDE.md rules 30–31):
  - Threat match → escalate to BLOCK
  - Clean → no change (never downgrades)
  - Timeout/error → add signal, keep existing verdict

AI URL model merge (AI_Model_Contracts.md, rules U1–U6):
  - aiUrlRiskHigh + allow  → escalate to INFORM (U1)
  - aiUrlRiskHigh + inform → escalate to WARN (U2)
  - aiUrlRiskHigh + warn   → keep WARN, add signal badge (U3)
  - aiUrlRiskHigh + block  → keep BLOCK, add signal only (U4)
  - aiUrlClean + any       → no change, internal signal only (U5)
  - AI unavailable          → no change (U6)

Backend merge:
  - If backend returns a usable assessment, merge GSB + AI on top
  - If backend unavailable, use local + GSB + AI result
  - Highest severity wins in all cases

================================================================================
FULL PIPELINE FLOW (V1 — TWO-STEP SAFETY FLOW)
================================================================================

Step 1 — Input and Check:

  User provides input via omnibox (type, paste, tap)
      ↓
  Input classifier: URL or search query?
      ↓ URL
  Normalize URL (scheme, host, domain parts, TLD, punycode, etc.)
      ↓
  Show checking spinner ("Checking link…" → "Still checking…" → timeout)
      ↓
  Run in parallel:
    (a) Local engine: signal extraction → policy engine → verdict
    (b) GSB Lookup API query
    (c) AI URL signal model (Pro only, if configured)
    (d) Backend reputation check (if configured)
      ↓
  Deterministic combiner: merge all results, highest severity wins
      ↓
  Map to UX: screen type, buttons, message, signal badges
      ↓ → proceed to Step 2

  Input classifier: search query?
      ↓
  Open search results in built-in browser. No verdict screen.

Step 2 — Verdict and Action:

  Show verdict screen. User MUST tap a button. No auto-proceed.
      ↓
  OK verdict:
    Two equal buttons: "Continue in LinkLook" / "Open in {browser}"
      ↓
  INFORM verdict:
    Large "Go Back" button (dominant)
    Medium "Check Message Context" button (Pro feature)
    Two small buttons: "Continue in LinkLook" / "Open in {browser}"
      ↓
  WARN verdict:
    Large "Go Back" button (dominant)
    Medium "Check Message Context" button (Pro feature)
    Two small buttons: "Continue in LinkLook" / "Open in {browser}"
      ↓
  BLOCK verdict:
    Large "Go Back" button only. No forward path.
    Always show the reason for blocking.
      ↓
  UNKNOWN verdict:
    "Go Back" only, with retry option.

WHAT LIVES WHERE (V1)
================================================================================

ON-DEVICE
- DecisionEngine (signal extraction + policy evaluation)
- URLNormalizer + InputClassifier
- PolicyConfig (loaded from bundled JSON)
- GSBLookupClient (Lookup API v4 via URLSession)
- GSBCache (5-minute TTL, max 500 entries)
- URL signal model (Core ML, Pro only, optional)
- Message signal model (Core ML, Pro only, optional)
- URLFeatureExtractor (derives model input from NormalizedURL)
- UXMapper (verdict → screen layout)
- History store (Recent screen)
- Browser preference store

OPTIONAL BACKEND
- Reputation lookup service
- Backend results merge with local (highest severity wins)
- Backend unavailability does not break the check flow

================================================================================
DETECTION CAPABILITY — INTERNAL DESIGN TARGETS
================================================================================

These are internal targets, not marketing claims.

Tier 1 (structural + GSB):  ~75–85% of genuinely malicious links.
Tier 2 (AI signals, Pro):   ~5–10% additional (subtle patterns rules miss).
Combined estimated detection: 80–90%.
For primary audience (elderly users, mass-market phishing): 85–92%.
False positive target: below 1% of safe links. Design ambition: below 0.5%.

The remaining 10–20% is weighted toward sophisticated zero-day attacks.
This boundary is stated honestly in onboarding and the App Store description.
